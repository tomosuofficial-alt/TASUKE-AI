## 1. 背景 / なぜ作るか

- よいどころ千福（飲食）の定例ミーティングで、各自のタスク進捗・決定事項の達成度・業務達成度を記入する運用が既にある。
- しかし会議では現場報告として「さらっと流れて」しまい、吸い上げた生データが活かされず放置されがち。
- 代表の要望（要約）: 出てきた未達・つまずきを追って「なぜできなかったか / どうしたらできるか / どこでつまずいているか / 人の問題か、仕組み・動線の問題か」を詰め、改善して再トライしたい（＝PDCA）。
- 代表の本音: 「自分たちでは宙ぶらりんになり、日々の業務に忙殺されてなぁなぁになる。ルール・マニュアル・仕組みを一つずつ潤滑に回るよう構築したい」。

---

## 2. ゴール / ノンゴール

**ゴール**
- 会議で出た未達・課題を「改善カード」として記録・追跡し、原因分析 → 改善 → 再トライ → 定着（ルール/マニュアル化）まで回す。
- 「人の問題」と「仕組み・動線の問題」を**分離して可視化**する（個人攻撃でなく仕組み改善に向かわせる）。
- 現場が**スマホで・低摩擦で**使える。
- 最終的に**店舗側が自走**でき、TOMOSU. は監査役に退ける。

**ノンゴール（やらないこと）**
- 個人評価・監視ツールにしない（現場の協力が得られなくなる）。
- 過剰実装をしない。Phase1 は「使われること」が最優先。
- TOMOSU. に永続依存する設計にしない。

## 3. 戦略的前提（役割と3フェーズ）

TOMOSU. は「運転手（代行）」ではなく「設計者」。システムは内製オーナーへ徐々に移譲できる設計であること。

| フェーズ | 期間目安 | TOMOSU. の動き | システムの役割 |
|---|---|---|---|
| 第1期 立上げ | 〜1–2ヶ月 | カード起票・原因分類・改善設計を主導 | 最小機能で回す。摩擦ゼロ重視 |
| 第2期 移管 | 〜3–4ヶ月 | 社内オーナーへ運用移管・伴走 | 入力主体が店長/幹部へ |
| 第3期 監査 | 以降継続 | 月次レビューで型の精度向上 | 集計・トレンド可視化 |

## 4. 既存資産 / 技術スタック

- 既存資産: **千福チェックシートアプリ**（別途開発項目として実在）、**Notion CRM**（千福用）、**棚卸システム**。

## 5. アーキテクチャ方針（配置の決定ポイント）

PDCA ループをどこに実装するか。3案。

| 案 | 内容 | 長所 | 短所 | 資産化(KGI) |
|---|---|---|---|---|
| A | 既存チェックシートアプリを拡張（Next.js/Supabase） | 自社プロダクト資産化・UX 自由・横展開可 | 実装コスト大 | ◎ |
| B | Notion で構築（既存 DB 活用） | 最速・低コスト・現場が既に使用 | プロダクト資産にならない・自由度低 | △ |
| C | ハイブリッド（Phase1=Notion で検証 → Phase2 でアプリ移行） | 早く回し、検証後に資産化 | 二度手間の可能性 | ○ |

**推奨: C → 最終 A。** Phase1 は Notion で素早く回して運用検証し、型が固まったらチェックシートアプリに統合してプロダクト資産化する（KGI＝資産化と、現場が早く回したい現実の両立）。
※ 最終判断は §10① で確認。

## 6. データモデル（中核）

中核エンティティ = **改善カード（Improvement Card / 課題カード）**。
会議で記録される進捗・達成度のうち、未達・一部達成・要改善のものをカード化する。

主要フィールド:

| field | 説明 | PDCA |
|---|---|---|
| `id` | 一意 ID | — |
| `source_meeting_date` | 発生会議日 | — |
| `reporter` | 報告者（誰の現場か） | — |
| `category` | 区分: タスク / 決定事項 / 業務 / 現場オペ | — |
| `title` | 項目名 | — |
| `description` | 内容・現状 | — |
| `achievement` | 達成度（5段階 or %） | Check |
| `status` | 未着手 / 進行中 / 検証中 / 完了 / 横展開済 | — |
| `cause_type` | **原因分類（最重要・後述）** | — |
| `why` | なぜできなかったか（自由記述） | — |
| `next_action` | どうしたらできるか | **Plan** |
| `action_owner` | 改善担当 | Plan |
| `due_date` | 期限 | Plan |
| `verification` | 検証結果（次サイクルで効果確認） | **Check** |
| `to_standardize` | 横展開フラグ（ルール/マニュアル化すべきか） | **Act** |
| `related_doc` | 関連マニュアル/ルールへのリンク | Act |
| `created_by` / `updated_at` | 監査用 | — |

**`cause_type`（原因分類）= 本システムの肝**。代表が明示的に求めた「人 vs 仕組み vs 動線」の切り分け:
- ① ヒューマンエラー（人）
- ② 仕組み・ルールの不備
- ③ 動線・オペレーション設計
- ④ 情報共有不足
- ⑤ スキル / 教育不足
- ⑥ ツール / 設備
- ⑦ その他

## 7. 運用フロー（ループ）

1. 会議で進捗・達成度を吸い上げ
2. 未達 / 要改善項目を改善カード化
3. `why` 記入 → `cause_type` 分類 → `next_action` / `action_owner` / `due_date` 設定（**Plan**）
4. 現場で実行（**Do**）
5. 翌週 / 翌会議で追跡、`verification` 記入（**Check**）
6. 達成 → `to_standardize` でルール/マニュアル化（**Act**）/ 未達 → 原因再分析して再トライ

サイクルは会議頻度に同期（週次想定。§10④ で確認）。

---

## 8. 機能要件（MVP）

- カードの CRUD（スマホ最適化フォーム）
- ステータス別ボード（かんばん: 未着手 / 進行中 / 検証中 / 完了 / 横展開済）
- 原因分類フィルタ・担当者フィルタ・期限ソート
- 期限超過のハイライト
- 横展開フラグ付きカードの「ルール/マニュアル化リスト」抽出

---

## 9. 出力 / ダッシュボード（経営インサイト）

- **原因分類の集計**（人 vs 仕組み vs 動線 … の比率）← 代表が最も欲しい気づき。「仕組み/動線」が多ければ個人でなく仕組みを直す、という判断材料になる。
- ステータス別件数 / 完了率 / 平均クローズ日数
- 横展開（ルール化）された件数 ＝ 仕組みが積み上がった量の指標
- 月次サマリー（第3期 監査フェーズ用）
