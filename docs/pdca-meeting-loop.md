# PDCA 会議ループ運用マニュアル
> 最終更新: 2026-06-05（役割分担改訂）/ 作成: マワリ（TASUKE.AI COO）
> 本運用用。試走なし。代表承認後すぐ動く。

---

## 30秒サマリー

**課題**: 定例会議でアクションを決めても「会議と会議のあいだ」で詰めず、次の会議でまた同じ課題が出る。

**解決策**: 会議直後に台帳化（拾う）→ 中間で未達だけ4問で深掘り（詰める）→ 次回会議冒頭に1枚で返す（返す）。このループを毎月回す。

**スコープ**: まず千福で建て付け。Niki★DINER / Bistro Knocks / Mz cafe へ横展開。

**重要な割り切り**: 仕組み原因のものだけ資産化する。ヒューマンエラーは仕組み化しない。

---

## 役割分担

AI（マワリ）が記録・分析の作業負荷を吸収し、大内さんの負担を判断と現場確認の3点に絞る。
「日々の業務に忙殺されてなぁなぁになる」を構造的に防ぐための設計。

| ステップ | 誰がやる | 内容 |
|---------|---------|------|
| ①拾う（台帳入力） | **マワリ** | 議事メモを受け取り、アクションをNotion台帳に1行ずつ入力 |
| ②詰める（4問深掘り） | **マワリ**（ドラフト）→ **大内**（○×判断） | マワリがドラフトを作成し、大内さんが内容を確認・判断 |
| ③返す（1枚シート整形） | **マワリ**（作成）→ **大内**（会議で使用） | マワリが達成チェックシートを整形し、大内さんは会議冒頭で見せるだけ |
| 現場実行確認 | **大内** | 店舗でアクションが実際に動いているかを確認 |

### 前提条件（必須）

①拾うと③返すをAIが代行するには、**議事メモを会議後すぐ共有してもらうことが必須**。
形式は問わない。録音の文字起こし・手書き写真・箇条書きメモのいずれでも対応する。
メモがなければ台帳入力の起点がなく、ループが動かない。

| ステップ | タイミング | 担当 | やること | 残す場所 |

## 2. アクション追跡台帳 — Notion設計

### 判断: 既存タスクDB（LINE吸い出し）を拡張 vs 新規DB

**→ 新規DBを作る（理由は下記）**

| 観点 | 既存タスクDB（LINE吸い出し） | 今回の新規DB |
|------|--------------------------|------------|
| 用途 | LINEの見落とし・依頼事項を拾う | 定例会議のアクションをPDCAで追う |
| 粒度 | Geminiが自動抽出した断片タスク | 会議で合意した約束事（担当・期限確定済み） |
| ライフサイクル | 発生→対応→閉じる（短期） | 月次ループで繰り越し・振り返り（月次） |
| 必要な列 | 種別・緊急度・動く人 | つまずき分類・会議回・資産化リンク（会議専用列が必要） |

既存DBに会議専用列を追加すると「LINEタスク」と「会議アクション」が混在して見通しが悪くなる。別DBにして用途を分けた方が運用が楽。

### DB名: 定例アクション追跡

**親ページ**: Notionトップ（`3249fe8c3a7f80269c41ce55e14d4d79`）配下

### 列定義

| 列名 | 型 | 選択肢 / 備考 |
|------|-----|------------|
| アクション | タイトル | 「たつや：ホール俯瞰・指示のみで回す」のような形式 |
| 担当者 | セレクト | ちい / たつや / ひろき / なぎさ / いくえ / あきお / おおうち / 全員 |
| 状態 | セレクト | 未着手 / 進行中 / 完了 / 未達 / 繰り越し |
| 期限 | 日付 | 会議で決めた期限日 |
| 対象クライアント | セレクト | よいどころ千福 / Niki★DINER / Bistro Knocks / Mz cafe |
| 対象会議回 | セレクト | 2026-04 / 2026-05 / 2026-06 … （毎月追加） |
| つまずき分類 | セレクト | 仕組み/動線 / ヒューマンエラー / 外部要因 / 知識/スキル不足 / 未分類 |
| なぜできなかったか | テキスト | ②詰めるで記入 |
| どこでつまずいたか | テキスト | ②詰めるで記入 |
| どうすればできるか（改善案） | テキスト | ②詰めるで記入 |
| 資産化リンク | URL | 仕組み原因のみ。decision-log.js で作成した意思決定ログのNotion URL |
| 備考 | テキスト | 達成根拠・補足など |

**つまずき分類の定義**:
- **仕組み/動線**: やり方・手順・ツール・確認フローが整っていなかった → 資産化対象
- **ヒューマンエラー**: 知っていたのに抜けた・うっかり → 資産化しない
- **外部要因**: 業者対応遅延・天候・急な人員変化など → 資産化しない（記録のみ）
- **知識/スキル不足**: やり方を知らなかった → 教育対応（マニュアル化検討）
- **未分類**: ②詰めるの前の初期状態

## 3. 「詰める」深掘りテンプレ（4問）

中間チェックのとき、マワリが「未達」「進行中」の行を1件ずつ開いてこの4問のドラフトを埋める。
大内さんは上がってきたドラフトに○×の判断を入れるだけ。1問ずつ、短い回答でいい。言い訳ではなく構造を探す。

---

### テンプレ

```
【深掘りシート】

対象アクション: _______________
担当: _______________  期限: _______________

Q1. なぜできなかったか？
→

Q2. どこでつまずいたか（最初に詰まった場所）？
→

Q3. これは「仕組み/動線の問題」か「ヒューマンエラー」か？
   → [ 仕組み/動線 / ヒューマンエラー / 外部要因 / 知識/スキル不足 ]
   判断根拠:

Q4. どうすればできるか（次月のアクションレベルまで落とす）？
→

つまずき分類: _______________
資産化する？ [ する / しない ]
```

### 千福の実データ記入例（2026-04 会議のアクション）

対象アクション: **たつや：ドリンク管理の継続**
担当: たつや  期限: 2026-04-30（3月中の継続課題）

Q1. なぜできなかったか？
→ 会議で「ドリンク管理を続ける」とだけ決め、「毎日いつ、誰が確認するか」を決めなかった。確認者が曖昧なまま放置された。

Q2. どこでつまずいたか？
→ 確認のタイミング（営業前 or 営業後）と記録場所（紙 or チェックリストアプリ）が決まっていなかった。始まらないまま月が終わった。

Q3. 仕組み/動線 か ヒューマンエラー か？
→ **仕組み/動線**。やる気はあったが「いつ・どこで・どう記録する」の動線がなかった。

判断根拠: チェックリストアプリに「ドリンク在庫確認」の項目がなかった。追加すれば自然に回る。

Q4. どうすればできるか？
→ チェックリストアプリに「ドリンク在庫確認（営業前17:00）」の項目を追加する。確認者はたつや。追加は大内が2026-06-07までに実施。

つまずき分類: **仕組み/動線**
資産化する？ **する**（意思決定ログ：「チェックリストアプリへのドリンク管理項目追加」として記録）

## 5. 仕組み資産化ルール

### 資産化する条件

「つまずき分類」が **仕組み/動線** のもの、かつ「同じ課題が別の担当者・別の月にも起きうる」もののみ資産化する。

### 資産化の手順

1. Notion台帳の「資産化する？」を「する」にチェック
2. `decision-log.js` を使って意思決定ログDBに1件追記する

```bash
node decision-log.js "チェックリストアプリにドリンク管理項目を追加" \
  --body "たつやのドリンク管理タスクが未達。チェックリストに項目がなかったため動線がなかった。項目を追加して毎日の確認を仕組み化する。" \
  --reason "仕組み/動線が原因と判定。ヒューマンエラーではない。" \
  --decision "チェックリストアプリの「ドリンク在庫確認（営業前17:00）」を追加。担当: たつや。" \
  --ai "COO/Gemini" \
  --priority 中
```

3. 台帳の「資産化リンク」列に、作成したNotionページのURLを貼る
4. 必要に応じて `docs/` 配下のマニュアルにも追記する（例: チェックリスト運用ルール）

### ヒューマンエラーを資産化しない理由

仕組み化は「誰がやっても同じ結果を出せる動線を作る」ことが目的。
ヒューマンエラー（知っていたのに抜けた）を仕組み化しようとすると、
ルールや確認ステップが無限に増えて運用が重くなる。
「気をつける仕組み」は実質的に「気をつけます」と同じ。

ヒューマンエラーへの対応は **個別フィードバック** にとどめ、台帳には記録するが資産化はしない。
