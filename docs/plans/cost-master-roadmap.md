# 原価計算 Web app 化に向けたロードマップ

## Context

TASUKE.AI の事業化候補として「飲食店向け原価計算 Web app」を構想中。
現在 Bistro Knocks 向けに作っている Excel 原価表は、その**論理設計図そのもの**になっている（食材→レシピ材料→サブレシピ→サイド→ソース→メニュー→税抜計算→原価率/利益率）。
飲食業の原価計算は手作業×Excelが業界標準で、痛みが明確かつ汎用性が高いため、TASUKE.AIのミッション（飲食店のAI活用支援）にも直結する。

ユーザー判断: **段階的アプローチ**を採用。先に1案件で骨を仕上げ、テンプレ→プロダクトと展開する。

## 全体ロードマップ（3 Phase）

### Phase 1: Bistro Knocks 原価表 完成（現在地・短期）

「完成」の定義 = 全25メニューの原価率と利益率が業者単価実値に基づいて表示され、クライアント納品できる状態。

未完了タスクの全量:

**A. 業者単価の収集と入力（残）**
- 食材マスタ: ハラミ／ロービー（または牛モモから歩留まり計算）／ミスジ／イベリコ／厳選和牛／鯛／サーモン／カジキ／赤魚／大豆バーグ／豆腐／とりむね
- レシピ材料マスタ: 味醂／醤油／マギーブイヨン／顆粒だし／しめじ／えのき／エリンギ／粒マスタード／白ワイン／クラム／生クリーム／玉ねぎ(個/g)／生姜／にんにく(株)／日本酒／上白糖／レモン／かんたん酢／塩／グラニュー糖／とうがらし／白粒こしょう／大根／人参／パプリカ／ヤングコーン／きゅうり／サラダ油／ケチャップ／りんご／にんにく(個)／チリペッパー／サニーレタス／水菜／白菜／紫キャベツ／キャベツ／もやし
- 飲料マスタ: pet_free／pet／pack の原価
- 資材マスタ: 容器（黒/新）／蓋（黒/新）／箸／ワサガード／帯／おしぼり（シーザードレッシングは入力済）
- サイドマスタ単価: 卵／ご飯／かぼちゃ／小エビ／有頭エビ（自家製ピクルス・サラダベース・蒸し野菜・ブロッコリーはレシピ計算と連動済）

**B. レシピ未受領**
- ポテトサラダのレシピ（材料・分量）

**C. ロジック未確定**
- 牛モモ → ロービー の歩留まり係数（仕入kg単価から提供gあたり原価への変換）
- かぼちゃ・小エビ・有頭エビ の1食あたり分量（個数または g）

**D. 仕上げ**
- デザイン整え（クライアント提出版の見た目）
- Googleスプレッドシート変換テスト（関数・書式が全部移行するか確認）
- クライアント（大内さん）に提出して承認

### Phase 2: テンプレート化（中期）

Bistro Knocks 完成後、以下を実施:

- `clients/_template/cost_master_template.xlsx` を抽出（Bistro Knocks固有データを汎用プレースホルダーに）
- ビルドスクリプト `build_knocks_cost.py` をジェネレーター化
  - 入力: YAML or JSON（食材リスト、レシピ、メニュー）
  - 出力: xlsx
- 業種別バリアント（弁当店／居酒屋／カフェ／レストラン）の差分把握
- 別クライアントで1案件 0→1 を試行（よいどころ千福 or Niki★DINER で適用）

### Phase 3: Web app化（中長期）

**差別化軸: 「AI で入力作業を消す」**

コア機能:
1. 納品書を写真で撮ると AIが自動で副資材マスタに登録
2. レシピを写真 or 自然言語で入力するとパースして材料マスタに登録
3. メニュー名と食材を入れると原価率の目安をAIが提案
4. 業者切替時の見積もり比較
5. Excel/CSV 入出力（既存ユーザー資産の取り込みと「データはユーザーのもの」原則）

技術スタック候補（後で詰める）:
- フロント: Next.js（TASUKE.AI既存スタック）
- データ: Supabase or Cloudflare D1
- AI: Claude API（OCR、レシピパース、原価率提案）
- POS連携: スマレジ（既存実装あり）
- 認証: Supabase Auth / Clerk

価格帯仮説:
- 月額 5,000〜15,000円/店舗
- 競合: TanaCoCo、HANJO、Smaregi原価管理、freee原価管理

差別化チェックポイント:
- 入力 UX で勝てるか（写真 → AI 自動入力の体験）
- 飲食店のITリテラシーに合うシンプルさを保てるか
- POS連携で「売上×原価率＝粗利」をリアルタイム表示まで届けられるか
